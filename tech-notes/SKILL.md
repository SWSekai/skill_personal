## 技術筆記整理

當對話中出現技術知識類問答時，主動整理成結構化的 markdown 技術筆記。

---

### 觸發條件

#### 模式 A：手動觸發（主要）
使用者明確下達指令並指定主題標題，例如：
- 「整理一篇 SQL JOIN 的技術筆記」
- 「把剛才的 Git rebase 整理成文章」

#### 模式 B：偵測提醒（被動）
當偵測到對話中出現技術知識型問答時，**不直接生成文件**，而是以訊息提醒：

> 這個主題可以整理成技術筆記：**{建議主題名稱}**，需要我整理嗎？

使用者確認後才執行。

### 輸出規範

1. **存放路徑**: `.local/docs/tech-notes/`
2. **檔名格式**: `{主題類別}_{具體主題}.md`（例：`sql_comments.md`、`git_rebase.md`）
3. **文章必須是完整結構化文章**，包含目錄、架構、範例：

```markdown
# {主題標題}

> 建立日期: YYYY-MM-DD | 最後更新: YYYY-MM-DD | 分類: {類別}

## 目錄
- [概述](#概述)
- [章節一](#章節一)
- ...

## 概述
這個技術是什麼、解決什麼問題、適用場景

## {章節一}
### {子章節}
語法說明 + 完整可執行的程式碼範例

## {章節二}
...

## 常見陷阱與注意事項
- 不同環境/資料庫的差異
- 容易犯的錯誤

## 最佳實踐
實務建議與推薦做法

## 延伸閱讀
相關主題與進階方向
```

4. **風格要求**:
   - 用繁體中文撰寫
   - 每個語法點都要附完整可執行的程式碼範例
   - 標註不同環境/資料庫的差異（如適用）
   - 文章要有深度，像一篇教學文章而非筆記片段
   - 包含由淺入深的結構，從基礎到進階

### 執行流程

1. 確認 `.local/docs/tech-notes/` 目錄存在
2. 檢查是否已有同主題文件 — 有則更新補充，無則新建
3. 撰寫完整結構化文章
4. 告知使用者文件路徑
