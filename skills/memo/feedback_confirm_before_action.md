**強制：以下任務在動作前必須先呈現計畫並等待確認**

適用範圍：
- 解析人工標注規則（如 audit 表的 `_a_` 分類）並批次套用
- 對多個檔案做相同邏輯處理（如 7 份 PDF 同樣腳本重生）
- 會覆蓋／改寫使用者既有手填內容（如重生 audit md）
- 修改 GT 等資料庫等級的真相來源（adding rows to GT/jsonl）
- 改動推論／訓練 pipeline 的關鍵參數

呈現格式：
1. **理解確認**：把使用者敘述的規則用具體例子重述（如「FAI 1,3,4,6 → 缺 2,5」）
2. **計畫拆解**：列出每個檔案、每個函式、每個步驟
3. **開放問題**：列出 2-4 個拍板點（閾值、優先級、命名等），可用 AskUserQuestion
4. **執行順序**：標明 1→2→3 與回報節點

收到使用者回覆「GO」或具體答覆後才開始實作。

**Why:** 使用者 2026-05-11 明確要求「每次都必須這麼做」。歷史教訓：批次腳本若未對齊預期，會覆蓋人工成果（如審查過的 `_a_` 分類）。

**How to apply:**
- 簡單單檔修改、單一明確問題 → 直接做，不需要走計畫流程
- 涉及上述適用範圍時 → 即使使用者沒明說，也要先呈現計畫
- 計畫接受後，每完成一個主要步驟回報數字（行數、筆數、變化），但不再重複問
- 中途若發現原假設錯誤、需改變方向 → 暫停、回報、再次確認

**與 Rule 15（不問 Y/N）的關係：**
這條規則不違反 Rule 15。計畫呈現的格式是「陳述句 + 拍板選項」，不是「要不要 X？」的問句。拍板選項用 AskUserQuestion 或編號列表（使用者直接回 1/2/3 或自由文字），不是「Y/N」。
