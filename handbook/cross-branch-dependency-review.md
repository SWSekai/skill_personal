# 跨分支依賴問題：審核者處理指南

> 適用情境：多位工程師並行開發不同功能，各自維護獨立分支，其中一個分支對另一個尚未 merge 的分支產生依賴。

## 問題分類

在處理跨分支依賴之前，審核者必須先判斷問題的性質：

```
跨分支依賴
├── 情境一：技術依賴（A 需要 B 的程式碼才能運作）
│   └── → 處理 merge 順序與分支策略
│
└── 情境二：需求缺口（A 的存在讓 B 的邏輯不完整）
    └── → 需求層面問題，必須 escalate
```

---

## 情境一：單純的跨分支技術依賴

### 問題描述

分支 A 的功能在程式碼層面依賴分支 B，但 B 尚未 merge 進主線。

### 處理選項

#### ✅ 方案一：先 merge 被依賴的分支（最推薦）

**適用條件**：B 分支已足夠成熟，通過 review。

**步驟**：
1. 優先 review 並 approve B
2. merge B 進主線
3. 讓 A 從主線 rebase，取得 B 的程式碼
4. 再 review 並 merge A

**優點**：分支歷史乾淨，無複雜依賴鏈  
**缺點**：若 B 未完成，A 的開發會被阻塞

---

#### ✅ 方案二：A 從 B 分支開出（或 cherry-pick）

**適用條件**：B 短期內無法 merge，但 A 需要繼續開發。

**步驟**：
1. A 的工程師從 B 分支建立新分支
2. 或 cherry-pick B 中需要的 commits
3. PR 描述中明確標注 `depends on #PR_B`
4. **強制要求 B 先 merge，再 merge A**

**優點**：A 可繼續開發，不阻塞  
**缺點**：B 有變動時，A 需要 rebase；merge 順序必須嚴格控制

---

#### ✅ 方案三：使用 Feature Flag 解耦

**適用條件**：兩個功能可獨立上線，但 A 的邏輯依賴 B 完成後才啟用。

**做法**：讓 A 透過 feature flag 隔離，兩者可獨立 merge，flag 關閉時 B 的缺失不影響系統。

---

#### ⚠️ 方案四：質疑設計，要求重構

**適用條件**：依賴關係暗示兩個功能的邊界劃分有問題。

**做法**：退回兩個 PR，要求工程師討論如何重新拆分職責，或抽出共用介面/模組。

> 跨分支依賴往往是**設計問題的信號**，審核者應有意識地識別這類情況。

---

### 情境一決策矩陣

| 情況 | 建議做法 |
|------|---------|
| B 即將完成 | 等 B merge 後，A rebase |
| B 還早，但需求明確 | A branch from B，標注依賴，控制 merge 順序 |
| 功能可獨立啟用 | Feature Flag 解耦 |
| 依賴關係疑似設計問題 | 退回，要求重新設計邊界 |

## 情境二：需求遺漏導致的下游流程缺口

### 問題描述

```
A 功能 ──觸發──▶ B 下游流程
                    ↑
              B 的需求規格當初未考慮到
              來自 A 的 input 情境
```

B 按照原始需求開發完畢，但 A 帶來的新情境讓 B 的邏輯不完整或行為未定義。

> ⚠️ **這不只是 code review 問題，而是需求規劃層面的漏洞。**

### 危險性說明

這類問題比「功能未完成」更危險：

- 系統**不會報錯**
- 但行為是**靜默錯誤**（silent bug）
- 難以在測試階段被發現

### 處理步驟

#### 第一步：釐清責任歸屬

這不是 A 工程師或 B 工程師的錯，是**需求規劃階段的漏洞**。

- 不要期待工程師自行協調解決
- 直接將問題 **escalate 給 PM / Tech Lead**
- 要求補上需求文件，明確定義 A 觸發 B 時的預期行為

---

#### 第二步：凍結 A 的 merge

在 B 的行為缺口被明確定義之前，**禁止 A merge 進主線**。

```
❌ 先 merge A，之後再補 B 的處理邏輯
✅ 先定義清楚，再 merge
```

---

#### 第三步：要求補充需求，釐清以下問題

```
1. A 觸發 B 時，B 應該如何處理？
   ├── 走新的流程分支？
   ├── 擴充既有流程邏輯？
   └── 拒絕 / 忽略此情境？

2. B 的原始邏輯在 A 的情境下是否仍然成立？

3. 是否需要為 B 補開新的 ticket？
```

---

#### 第四步：在 B 的 PR 明確標注缺口

在 PR 留下評論，例如：

```
⚠️ 此 PR 未處理來自 A 功能的觸發情境（見 PR #xxx）。
在 A 的需求補充並實作之前，建議不要 merge A。
需要 PM 確認 B 在此情境下的預期行為。
```

---

### 情境二處理流程

```
發現需求缺口
     │
     ▼
凍結 A 的 merge
     │
     ▼
Escalate 給 PM / Tech Lead，要求補需求
     │
     ▼
確認新需求後，要求 B 補開 ticket 或在現有 PR 處理
     │
     ▼
兩個 PR 皆完整後，控制順序 merge
```

## 常見錯誤做法

| 錯誤做法 | 為什麼不好 |
|---------|-----------|
| 叫 A 工程師自行去跟 B 協調 | 兩人可能各自妥協出不符需求的方案 |
| 讓 B 工程師自行「猜測」A 的行為 | 假設不一致，日後難以維護 |
| 先 merge A，之後再說 | 製造未定義行為，增加 debug 成本 |
| 把 B 的補充需求塞回 backlog | 問題被延遲，但 A 已上線造成潛在 bug |
| 叫工程師直接在 code 裡加 workaround | 掩蓋問題，累積技術債 |

## 審核者行動清單

### 技術依賴（情境一）

```
□ 確認依賴方向（A → B，還是互相依賴）
□ 評估 B 分支的成熟度
□ 選擇適當的分支策略（等待 / branch from B / feature flag）
□ 要求 PR 描述中明確標注 "depends on #xxx"
□ 設定 CI 保護規則，防止錯誤的 merge 順序
□ 若懷疑是設計問題，拉兩位工程師一起討論
```

### 需求缺口（情境二）

```
□ 確認 B 在 A 的情境下行為是否有明確定義
□ 凍結 A 的 merge
□ Escalate 給 PM / Tech Lead
□ 在 B 的 PR 留下清楚的警告評論
□ 追蹤補充需求的進度，確保不被遺忘
□ 確認需求補齊後，再重新 review 兩個 PR
```

---

## 核心原則

> **不要讓隱性依賴存在。**  
> 所有跨分支依賴都必須**顯性化**、**文件化**，並控制 **merge 順序**。

> **審核者不只是看程式碼對不對，而是要看見系統層面的缺口。**  
> 在問題進入 production 之前，在需求層面就攔截它，是資深審核者最重要的價值之一。
