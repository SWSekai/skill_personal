決策摘要生成規則延伸：

**區分選項類型：**

| 選項類型 | 特徵 | 摘要處理 |
|---|---|---|
| **單一路線（mutually exclusive）** | 選 A 就不能選 B（如 `/build commit` vs `/commit-push` 改名方案） | 只記錄最終選擇，未選項丟棄 |
| **非單一路線（non-exclusive / 功能選用）** | 多個選項可共存、可分階段採納（如 feature flags、附加功能、可選強化） | **保留所有未選項作為「備用清單」** 於摘要 |

**Why:** 使用者於 2026-04-15 Rule 17 決策流程對話中指出：有些決策的選項並非互斥，可能當下不選但未來仍有價值（例如「是否新增 /build review 子命令」、「是否修補 sp-init.bat Step 8b」這類功能選用題）。若摘要只記最終選擇會喪失這些未來可用的候選，等於每次重新評估都要從零開始。

**How to apply:**

1. **摘要生成時的判斷**：每個決策區塊的選項列表，先判斷「若全選是否會互相矛盾」
   - 會矛盾 → 單一路線 → 只記最終選
   - 不會矛盾 → 非單一路線 → 全部保留
2. **非單一路線的摘要格式**：
   ```markdown
   ## §X.Y 決策：<主題>
   - ✅ 已採納：<選項 A>
   - ✅ 已採納：<選項 B>
   - 🔖 保留候選（未採納但可重啟）：
     - <選項 C> — 理由：<為何這次不選、什麼情境下可能會選>
     - <選項 D> — 理由：...
   ```
3. **日後回查時機**：新對話提出類似需求 → 先讀 summaries/ 對應區塊 → 若有保留候選 → 直接評估是否重啟，不要重新設計
4. **候選清理**：保留候選若超過 6 個月未被重啟，或明確被新決策取代 → 於該次決策的摘要中標註「已作廢」，不要刪除（保留歷史）

**典型情境**：
- 功能選用：新增子命令、功能旗標、附加 UI 元素
- 技術選型備援：A 方案 / B 方案 / C 方案 並非互斥，只是當下選 A 最優
- 漸進採納：「現在先做核心，進階功能保留」

**不適用情境**（即仍走單一路線）：
- 架構二選一：選了 monorepo 就不能同時 polyrepo
- 命名衝突：同一個命令只能有一個定義
- 相互排斥的設定：嚴格模式 vs 寬鬆模式
